Ranked Ideas

2 ideas selected after user iteration. Caching, CI/CD (using pre-commit), environment switching, and plugin architecture removed.

1 Test Coverage for TUI Optimization Enabler

目标： 为后续 TUI 优化提供安全保障

Build test infrastructure (HTTP mock + TUI model tests) specifically to enable safe TUI refactoring and improvements. Current logs TUI has complex state (viewMode, navigation, async loading, highlighting) with no tests — making changes risky.

direct Basis: Only one test file exists (config_test.go with 3 tests). The logs TUI model (cmd/logs.go, 740-1273 lines) has no unit tests. Recent fixes (viewMode init, timezone, row limits) were manual — tests would have caught regressions.

Rationale: Tests are not the goal — they are the enabler. With test coverage, future TUI improvements (better scrolling, smoother animations, more views) can be made with confidence. Without tests, each TUI change is risky.

Downsides: Upfront investment before seeing TUI improvements.

Confidence: 90% Complexity: Medium

2 TUI-First with Optional JSON Output Inversion

Keep beautiful TUI as the default for human users. Add a --json flag for programmatic consumption (CI/CD pipelines, scripts). This inverts the common "JSON first" pattern.

direct Basis: The codebase has both TUI (NewLogsModel) and text mode (runLogsText) but text mode is hidden as fallback. No CLI flag exposes JSON output.

Rationale: Default UX stays beautiful TUI for humans. Automation scripts get --json for parseable output without ANSI escape codes. Best of both worlds.

Downsides: Need to ensure JSON output covers all commands consistently.

Confidence: 80% Complexity: Medium
